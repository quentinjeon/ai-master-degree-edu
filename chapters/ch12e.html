RAG — 검색 증강 생성

LLM이 모르는 것을 찾아서 답하게 하는 법 — Retrieval-Augmented Generation의 수학과 아키텍처. 논문·기업 AI 시스템의 90%가 사용하는 핵심 기술.

🧒 초등학생 한 줄 설명

"오픈북 시험과 클로즈북 시험의 차이다. LLM이 혼자 기억에 의존하면 클로즈북(오래된 정보, 환각 가능). RAG는 시험장에 교재를 가져와서 답을 찾아 쓰는 오픈북 시험이다."

  • 클로즈북 LLM: "2026년 최신 논문에 대해 알려줘" → 학습 데이터에 없어서 환각(hallucination)
  • RAG: 질문 → 벡터 DB 검색 → 관련 논문 조각 가져오기 → LLM이 보며 답변
  • 기업 활용: 사내 문서 RAG, 법률 문서 RAG, 의료 기록 RAG
  • AI에서: ChatGPT 웹 검색, Perplexity AI, NotebookLM 모두 RAG 방식

🔍RAG 기본 파이프라인 — 검색 + 생성

RAG는 LLM의 두 가지 핵심 문제를 해결합니다: ①학습 데이터 이후 정보는 모름(지식 최신성), ②틀린 정보를 확신하며 생성(환각). 외부 문서를 검색해서 컨텍스트로 제공하면 두 문제가 크게 완화됩니다.

1
문서 전처리 (오프라인) — 문서를 청크(chunk)로 나눔 → 임베딩 모델로 벡터 변환 → 벡터 DB(FAISS, Pinecone 등)에 저장
2
검색 (실시간) — 사용자 질문을 동일한 임베딩 모델로 벡터 변환 → 벡터 DB에서 Top-k 유사 청크 검색
3
생성 (실시간) — 검색된 청크 + 사용자 질문을 프롬프트로 결합 → LLM이 참고하여 최종 답변 생성
RAG 생성 확률 (Lewis et al., 2020)
\[P(y \mid x) = \sum_{z \in \text{top-}k(p_\eta(\cdot|x))} p_\eta(z \mid x) \cdot p_\theta(y \mid x, z)\]
\(x\): 입력 질문
\(z\): 검색된 문서 조각(passage)
\(p_\eta(z \mid x)\): 검색 모델이 질문 \(x\)에 대해 문서 \(z\)를 선택할 확률
\(p_\theta(y \mid x, z)\): LLM이 질문 \(x\)와 문서 \(z\)를 보고 답 \(y\)를 생성할 확률
전체 확률: top-k 문서들에 대한 가중 합산 — 여러 근거 문서를 종합하여 답변 생성
📄

원본 논문: Lewis et al. (2020) "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks" (NeurIPS 2020) — RAG라는 용어를 처음 정립한 논문. Facebook AI Research에서 발표, 현재 인용 수 수천 회를 넘는 핵심 레퍼런스.

🧲Dense Retrieval — 벡터 유사도로 의미 검색

전통적 키워드 검색(BM25)은 "단어가 겹치는지"만 봅니다. Dense Retrieval(DPR)은 임베딩 모델로 질문과 문서를 같은 벡터 공간에 표현하여 의미적 유사도로 검색합니다. "자동차"와 "차량"처럼 표현이 달라도 의미가 같으면 검색됩니다.

DPR 유사도 점수 (Karpukhin et al., 2020)
\[s(q, p) = E_Q(q)^\top E_P(p) \quad\text{검색:}\quad \hat{p} = \arg\max_{p \in \mathcal{C}} E_Q(q)^\top E_P(p)\]
\(E_Q\): 질문 인코더(예: BERT 기반), \(E_P\): 문서 인코더
코사인 유사도 대신 내적(dot product) 사용 → 정규화된 임베딩에서는 동일
\(\mathcal{C}\): 전체 문서 코퍼스 (수백만~수억 개 문서)
MIPS(Maximum Inner Product Search): FAISS로 근사 검색 → \(O(\log n)\) 복잡도
두 인코더를 분리하여 문서 임베딩을 미리 계산·저장 → 실시간 검색 시 질문만 인코딩

BM25 vs Dense Retrieval 비교

항목 BM25 (희소 검색) Dense Retrieval (밀집 검색)
원리 키워드 매칭, TF-IDF 임베딩 벡터 유사도
동의어 처리 ❌ "자동차"≠"차" ✅ 의미적 유사도 포착
학습 필요 ❌ (통계 기반) ✅ (임베딩 모델 학습)
속도 매우 빠름 (역색인) 근사 검색 필요 (FAISS)
현재 추세 Hybrid(BM25+Dense)에서 병행 Dense가 대세, 단독 또는 Hybrid

🏗Advanced RAG 기법 — 검색 품질 향상

Naive RAG는 "그냥 Top-k 검색 후 붙여넣기"입니다. 실제 시스템에서는 검색 전·중·후를 모두 개선하는 Advanced RAG 기법이 필요합니다. 특히 전문 도메인(의학, 법률, 금융)에서 검색 품질이 답변 품질의 병목이 됩니다.

🔑 Advanced RAG 핵심 기법
  • Query Rewriting: 사용자 질문을 검색에 최적화된 형태로 변환 (검색 전)
  • HyDE (Hypothetical Document Embeddings): 가상 답변 문서를 먼저 생성 → 그 임베딩으로 검색 (검색 전)
  • Reranking: Cross-Encoder로 Top-k 결과를 재순위화 → 더 관련성 높은 문서 선택 (검색 후)
  • Chunk Size Tuning: 청크 크기·오버랩 최적화 → 문맥 유지와 정밀도 균형
  • Hybrid Search: BM25 + Dense + RRF(Reciprocal Rank Fusion)로 통합 → 두 방법의 장점 결합
HyDE — 가상 문서 임베딩 (Gao et al., 2022)
\[\hat{p} = \arg\max_{p \in \mathcal{C}} E_P(p)^\top E_P(\hat{d}) \quad\text{where}\quad \hat{d} \sim \text{LLM}(x)\]
\(x\): 사용자 질문, \(\hat{d}\): LLM이 생성한 가상 답변 문서
질문 \(x\) 대신 가상 답변 \(\hat{d}\)의 임베딩으로 검색 수행
가상 답변은 틀려도 됨 — 검색 공간에서 올바른 문서와 더 가까운 위치에 있으면 충분
전문 도메인(의학, 법률)에서 특히 효과적: 질문보다 가상 답변이 문서 스타일과 유사
단점: LLM 호출이 검색 전에 추가되므로 지연 시간 증가
Reciprocal Rank Fusion (RRF) — Hybrid 검색 점수 통합
\[\text{RRF}(d) = \sum_{r \in R} \frac{1}{k + \text{rank}_r(d)} \quad (k = 60 \text{ 권장})\]
\(R\): 여러 검색 방법의 집합(BM25, Dense 등)
\(\text{rank}_r(d)\): 방법 \(r\)에서 문서 \(d\)의 순위 (1위 = 1)
\(k = 60\): 상위 순위의 영향을 완화하는 상수 (경험적 최적값)
점수 스케일이 달라도 순위만 사용하므로 자연스럽게 통합 가능
구현 단순성 대비 효과 우수 — 대부분의 프로덕션 시스템에서 기본 선택

📊RAGAS — RAG 시스템을 어떻게 평가하는가

RAG 시스템은 일반 LLM 평가와 다릅니다. 검색 품질과 생성 품질을 따로 평가해야 합니다. RAGAS(RAG Assessment)는 사람의 레이블 없이 LLM 자체를 평가자로 활용하는 자동 평가 프레임워크입니다.

메트릭 평가 대상 측정 내용
Faithfulness 생성 품질 답변이 검색된 문서에 근거하는가 (환각 탐지)
Answer Relevancy 생성 품질 답변이 질문에 관련성 있는가
Context Precision 검색 품질 검색된 문서 중 실제 도움이 된 비율
Context Recall 검색 품질 필요한 정보가 검색에 포함되었는가
⚠️

환각(Hallucination)의 두 유형: ①Intrinsic: 검색된 문서와 모순되는 내용 생성, ②Extrinsic: 검색 문서에 없는 내용을 지어냄. RAG는 특히 Intrinsic 환각을 Faithfulness 메트릭으로 감지합니다. Top-k 결과가 충분히 관련성 있더라도 LLM이 이를 무시하고 사전 지식을 사용하면 Faithfulness가 낮아집니다.

📌 검색 품질 개선 방향
  • Context Precision 낮음 → Reranker 추가
  • Context Recall 낮음 → Top-k 증가, Chunk 크기 조정
  • 두 메트릭 모두 낮음 → 임베딩 모델 교체
  • Hybrid Search로 희소·밀집 검색 결합
📌 생성 품질 개선 방향
  • Faithfulness 낮음 → 프롬프트 강화 ("문서만 참조")
  • Answer Relevancy 낮음 → Query Rewriting 적용
  • 두 메트릭 모두 낮음 → LLM 모델 교체 or 파인튜닝
  • Citation 추가로 근거 투명성 확보
