📐벡터 공간에서 의미는 어디에 있는가

임베딩 모델은 텍스트를 고차원 벡터로 변환합니다. 잘 학습된 임베딩 공간에서는 의미적으로 유사한 텍스트가 기하학적으로 가까운 위치에 있습니다. 이 기하학적 구조가 RAG 검색의 수학적 토대가 됩니다.

코사인 유사도와 내적의 관계
\[ \cos\theta(u, v) = \frac{u \cdot v}{\|u\| \|v\|} \quad\Leftrightarrow\quad u \cdot v \quad\text{(정규화된 경우 동일)} \]
\(\|u\| = \|v\| = 1\) (L2 정규화)이면 코사인 유사도 = 내적
DPR 등 대부분의 검색 모델은 L2 정규화된 임베딩 사용
코사인 유사도: 방향 유사도(크기 무관) — 문서 길이 차이를 무시
유클리드 거리: \(\|u-v\|^2 = 2 - 2\cos\theta\) (정규화된 경우) → 동치 관계
임베딩 공간의 차원의 저주 (Curse of Dimensionality)
\[ \mathbb{E}\!\left[\frac{\max_{i} d(q, p_i) - \min_{i} d(q, p_i)}{\min_{i} d(q, p_i)}\right] \xrightarrow{d\to\infty} 0 \]
고차원 공간에서 모든 점 사이의 거리가 점점 비슷해지는 현상
768차원 임베딩: 여전히 차별적 신호 존재하지만 주의 필요
해결 전략: 차원 축소(PCA), 이분법적 인덱스(IVF), 근사 검색(ANN)
\(d(q, p_i)\): 쿼리 \(q\)와 문서 \(p_i\) 사이의 거리 / \(d\): 임베딩 차원
ℹ️

정규화의 중요성: 검색 전 L2 정규화(\(\|e\| = 1\))를 수행하면 코사인 유사도와 내적이 동치가 되어, FAISS의 내적 인덱스(IndexFlatIP)로 코사인 검색을 수행할 수 있습니다. DPR과 대부분의 임베딩 모델이 이 방식을 사용합니다.

⚡MIPS — 수억 벡터에서 가장 가까운 것 찾기

MIPS(Maximum Inner Product Search)는 쿼리 벡터와 내적이 가장 큰 벡터를 찾는 문제입니다. Exact MIPS는 \(O(n \cdot d)\)이므로 수억 개 문서에는 비실용적입니다. FAISS는 근사(Approximate Nearest Neighbor) 방법으로 이를 해결합니다.

IVF (Inverted File Index) — 군집 기반 근사 검색
\[ \hat{p} = \arg\max_{p \in \mathcal{C}_{k^*}} E_Q(q)^\top E_P(p) \quad\text{where}\quad k^* = \arg\min_{k} \|E_Q(q) - c_k\|_2 \]
\(c_k\): \(k\)번째 군집의 중심 벡터(k-means로 사전 계산)
쿼리와 가장 가까운 \(\text{nprobe}\)개 군집 내에서만 검색
\(\text{nprobe}\uparrow\) → 정확도↑, 속도↓ (trade-off)
전체 검색 대비 \(\text{nprobe}/\text{nlist}\) 비율만큼 검색 범위 축소
PQ (Product Quantization) — 벡터 압축
\[ E_P(p) \approx [q_1, q_2, \ldots, q_M] \quad\text{where}\quad q_m = \arg\min_{c \in \mathcal{C}_m} \|E_P(p)_{d_m} - c\|_2 \]
768차원 벡터를 \(M\)개 서브공간으로 분할 (예: \(M=8\) → 각 96차원)
각 서브공간에서 k-means → 코드북(codebook) 생성
원본 \(768 \times 4\) byte = 3072B → PQ 압축 후 \(8 \times 1\) byte = 8B (384배 압축!)
속도↑↑, 메모리↓↓, 정확도 약간 손실 — 수억 문서 RAG에서 필수

🎯DPR 학습 — 올바른 임베딩 공간 만들기

Dense Passage Retrieval(DPR)은 질문 인코더 \(E_Q\)와 문서 인코더 \(E_P\)를 대조 학습(Contrastive Learning)으로 훈련합니다. 정답 문서는 가까이, 관련 없는 문서는 멀리 밀어내는 방식으로 임베딩 공간을 최적화합니다.

DPR 학습 손실 — In-Batch Negative Sampling
\[ \mathcal{L}_{\text{DPR}} = -\log\frac{e^{s(q_i, p_i^+)}}{e^{s(q_i, p_i^+)} + \sum_{j \neq i} e^{s(q_i, p_j^-)}} \]
\(s(q,p) = E_Q(q)^\top E_P(p)\): 유사도 점수 (내적)
\(p_i^+\): 질문 \(q_i\)에 대한 정답 문서 (positive)
\(p_j^-\): 다른 질문의 정답 문서 (in-batch negatives) — 같은 배치의 나머지 문서들
배치 크기 \(B\)이면 \(B-1\)개의 negative 자동 생성 → 효율적 학습
Hard Negative Mining: 모델이 헷갈리는 어려운 negative 추가 → 성능↑
✅

최신 임베딩 모델: OpenAI text-embedding-3-large (3072차원), Voyage AI, Cohere Embed v3 등이 DPR의 원리를 대규모 데이터로 확장한 것입니다. In-Batch Negative를 수천 개 단위로 확장하고 Hard Negative Mining을 체계적으로 적용해 범용 임베딩 성능을 달성했습니다.

🏗Modular RAG — 현대 RAG 시스템의 구조

Self-RAG — 검색 여부 자체 결정 (Asai et al., 2023)
\[ P(y, r \mid x) = P(r \mid x) \cdot P(y \mid x, r, d^*) \quad\text{where}\quad d^* = \arg\max_{d} P(d \mid x) \cdot P(\text{isrel} \mid x, d) \]
\(r\): 검색 토큰 (Retrieve / No-Retrieve 중 모델이 결정)
항상 검색하는 Naive RAG 대신, 검색이 필요할 때만 검색
\(P(\text{isrel}|x,d)\): 검색된 문서가 관련있는지 모델이 스스로 판단
4개 특수 토큰 사용: [Retrieve], [IsRel], [IsSup], [IsUse]
이 토큰들로 검색 필요성·관련성·지지성·유용성을 단계적으로 평가
⚠️

Sparse vs Dense 검색의 상호보완: BM25(Sparse)는 정확한 키워드 매칭에 강하고, Dense는 의미적 유사성에 강합니다. 실제 프로덕션 RAG 시스템은 두 가지를 결합한 Hybrid 검색(RRF, Reciprocal Rank Fusion)을 사용하는 경우가 많습니다.
