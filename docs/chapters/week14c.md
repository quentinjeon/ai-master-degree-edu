## EASY BOX 내용

**핵심 비유**: "탐정은 단서를 보고 '이것이 사실이다 / 이렇게 해석된다 / 내 생각에는'을 구분한다. 연구자도 똑같이 논문을 읽는다."

**예시 리스트**:
- 사실: "실험에서 A 모델의 정확도는 82%였다" → 측정된 숫자, 반박 불가
- 해석: "A 모델은 B보다 문맥을 더 잘 이해한다" → 결과로부터 도출, 반박 가능
- 의견: "A 모델이 더 우수하다" → 주관적 판단, 기준 없으면 논증 불가
- AI에서: 논문 리뷰어는 이 세 가지를 섞어 쓴 논문을 가장 강하게 비판한다

## 섹션 구성

### 섹션 1: 사실, 해석, 의견 — 세 가지 층위

**아이콘**: `🔍`
**제목**: `연구자의 첫 번째 능력 — 세 층위를 구분한다`

**설명**: 논문을 읽거나 쓸 때 가장 많이 하는 실수는 해석을 사실처럼, 의견을 해석처럼 쓰는 것입니다. 레벨 3 리뷰어가 가장 먼저 지적하는 항목이기도 합니다. 세 층위를 구분하면 논문이 얼마나 강한 주장을 하는지 즉시 파악됩니다.

**표 (세 층위 정의)**:

| 층위 | 정의 | 특성 | 논문에서 위치 |
|------|------|------|-------------|
| **사실 (Fact)** | 측정·관찰된 것. 반박하려면 데이터가 필요 | "~였다", "~로 측정됐다" | Results 표, 실험 수치 |
| **해석 (Interpretation)** | 사실로부터 도출한 설명. 반박 가능 | "~임을 보여준다", "~를 의미한다" | Discussion 첫 문단 |
| **의견 (Opinion)** | 주관적 판단. 기준 없으면 검증 불가 | "~이 더 낫다", "~가 중요하다" | 해석 없이 쓰면 약점 |

**callout (warn)**:
- 내용: **의견을 해석처럼 쓰지 않도록 주의한다.** "이 방법이 더 중요하다"는 의견이다. "이 방법이 기존 대비 F1 5% 향상을 보여 실용적으로 유리하다"는 해석이다. 리뷰어는 이 차이를 반드시 잡는다.

### 섹션 2: 인과와 상관 — 가장 흔한 논리 함정

**아이콘**: `⚡`
**제목**: `"함께 움직였다"는 "만들었다"가 아니다`

**설명**: 두 변수가 같이 움직이는 것(상관관계)과 하나가 다른 하나를 만드는 것(인과관계)은 완전히 다릅니다. 이를 혼동하면 논문의 핵심 주장이 무너집니다. ML 논문에서 가장 자주 리뷰어에게 지적받는 항목 중 하나입니다.

**수식 박스 1**:
- 레이블: `상관계수와 인과의 차이`
- 수식:
```
\text{상관}: r(A, B) \neq 0 \quad\not\Rightarrow\quad A \to B
\text{인과}: A \to B \iff P(B \mid do(A)) \neq P(B)
```
- 주석:
  - `r(A,B)`: 피어슨 상관계수. 두 변수가 함께 변하는 정도
  - `do(A)`: 개입(intervention). A를 강제로 바꿨을 때
  - 상관은 관찰, 인과는 실험 또는 구조적 증명이 필요
  - ML 논문: "정확도가 높은 모델은 더 두꺼운 레이어를 가진다" → 상관. "레이어를 두껍게 하면 정확도가 오른다" → 인과 주장 (ablation 필요)

**표 (상관 vs 인과 판단표)**:

| 상황 | 상관인가 | 인과인가 | 판별 방법 |
|------|:-------:|:-------:|---------|
| A와 B가 같이 오르내린다 | ✅ | ❌ 가능성만 | 통제 실험 필요 |
| A를 바꿨더니 B가 바뀐다 (ablation) | — | ✅ 가능성 높음 | 다른 변수 통제 확인 |
| A, B 모두 C 때문에 오른다 | ✅ | ❌ 허위 상관 | 교란 변수 C 확인 |
| A를 고정하면 B가 변한다 | — | ✅ | RCT 또는 IV 설계 |

**callout (tip)**:
- 내용: **Ablation Study가 왜 중요한가.** "이 모듈을 빼면 성능이 떨어진다" = 이 모듈이 성능에 인과적으로 기여한다는 증거. 이것이 인과를 주장하는 실험적 방법이다.

### 섹션 3: 반례 먼저 찾기 — 가설 강화의 기술

**아이콘**: `⚔`
**제목**: `자기 가설에 먼저 반례를 찾는 사람이 강한 논문을 쓴다`

**설명**: 좋은 연구자는 자신의 주장이 틀릴 가능성을 먼저 생각합니다. 반례를 찾으면 두 가지 결과 중 하나입니다: 주장이 수정되거나, 주장이 더 강해집니다. 리뷰어가 제기할 반론을 미리 막는 것이 Discussion의 핵심입니다.

**표 (반례 찾기 4가지 질문)**:

| 질문 | 의미 | 논문에서 적용 |
|------|------|-------------|
| 데이터셋 문제인가? | 결과가 특정 데이터셋에만 나타나는 것 아닌가 | 다양한 데이터셋 실험 필요 |
| Baseline이 약한가? | 비교 대상이 최신 방법이 아닌 것 아닌가 | SOTA baseline 포함 |
| 특정 조건에만 좋은가? | 분포가 달라지면 결과도 달라지는 것 아닌가 | Distribution shift 실험 |
| 우연인가? | 반복 실험에서도 같은 결과가 나오는가 | Multiple seeds, CI 포함 |

**수식 박스 2**:
- 레이블: `반례로부터 가설 강화`
- 수식:
```
H \text{ (가설)} \overset{\text{반례 탐색}}{\longrightarrow}
\begin{cases}
H' \text{ (수정된 가설)} & \text{if 반례 발견} \\
H + \text{제한 조건 명시} & \text{if 반례가 가설 경계 내}
\end{cases}
```
- 주석:
  - 가설을 버리는 게 아니라 정교하게 만드는 과정
  - 좋은 논문: "우리 방법은 X 조건에서 효과적이고 Y 조건에서는 한계가 있다"
  - 나쁜 논문: 모든 조건에서 무조건 좋다고 주장

### 섹션 4: 측정 가능한 주장만 남긴다

**아이콘**: `📏`
**제목**: `측정 불가 주장은 논문에서 삭제한다`

**설명**: 연구자가 가장 많이 하는 실수 중 하나는 "측정할 수 없는 주장"을 논문에 그대로 쓰는 것입니다. 모든 주장은 실험, 수치, 논리 중 적어도 하나로 뒷받침돼야 합니다.

**표 (측정 불가 → 측정 가능 변환)**:

| 측정 불가 주장 (삭제 대상) | 측정 가능 주장 (논문 허용) |
|----------------------|----------------------|
| "이 모델은 더 효율적이다" | "이 모델은 동일 성능에서 50% 적은 FLOP을 사용한다" |
| "안전성이 향상됐다" | "정책 위반율이 기준 모델 대비 31% 감소했다" |
| "사용자가 더 만족한다" | "CSAT 점수가 3.8 → 4.2 (5점 만점) 향상됐다" |
| "학습이 더 빠르다" | "동일 성능 도달 시 epoch 수가 40% 감소했다" |
| "일반화가 잘 된다" | "OOD 데이터셋 3개에서 5% 이내 성능 드롭" |

**callout (info)**:
- 내용: **용어 정의는 논문 첫 등장 시 반드시 한다.** "효율성", "안전성", "공정성" 같은 단어는 분야마다 다르게 정의된다. 논문 안에서 "이 논문에서 X는 Y로 정의한다"가 없으면 리뷰어 지적 대상이다.

### 섹션 5: 대안 설명과 한계 — Discussion의 핵심

**아이콘**: `💡`
**제목**: `내 설명 말고 다른 설명 가능성을 쓴다`

**설명**: Discussion 섹션은 단순히 결과를 재설명하는 곳이 아닙니다. 결과에 대한 대안 설명, 연구의 한계, 일반화 가능성을 솔직하게 쓰는 곳입니다. 이것을 잘 쓸수록 논문이 신뢰를 얻습니다.

**표 (Discussion 4구조)**:

| 구조 | 내용 | 예시 |
|------|------|------|
| **결과 해석** | 왜 이 결과가 나왔는가 | "정책 레이어 추가가 위반율을 줄인 이유는..." |
| **대안 설명** | 다른 원인 가능성 | "단, 데이터셋 특성 때문일 가능성도 있다. 이를 확인하기 위해..." |
| **한계 명시** | 이 연구가 증명하지 못한 것 | "본 연구는 단일 도메인에서만 검증했으므로..." |
| **향후 연구** | 남은 질문 | "멀티도메인 설정에서 검증이 필요하다" |

**callout (tip)**:
- 내용: **한계를 쓰는 것은 논문을 약하게 만들지 않는다.** 오히려 리뷰어가 "이 연구자는 자기 연구의 경계를 정확히 안다"고 판단하여 신뢰도가 올라간다. 한계를 숨기면 리뷰어가 직접 찾아서 reject 사유로 쓴다.

## 인터랙티브: 주장 분류기 + 인과/상관 판단기

**제목**: `🔢 주장의 논리 수준 진단 — 사실 / 해석 / 의견 / 측정가능 판단`

**파라미터**: 드롭다운 선택 + 판단 결과 출력

**JavaScript 로직**:
```javascript
const claims = [
  {
    text: '"우리 모델의 정확도는 89.3%였다."',
    type: 'fact',
    causal: null,
    feedback: '✅ 사실 — 수치로 측정된 관찰. 반박하려면 데이터가 필요하다.'
  },
  {
    text: '"이 결과는 우리 모델이 문맥 이해 능력이 뛰어남을 보여준다."',
    type: 'interpretation',
    causal: null,
    feedback: '🔵 해석 — 사실(정확도 수치)로부터 도출한 설명. "문맥 이해 능력"이라는 해석은 반박 가능하다. 정의와 근거가 필요하다.'
  },
  {
    text: '"따라서 우리 방법이 가장 우수하다."',
    type: 'opinion',
    causal: null,
    feedback: '🔴 의견 — 측정 기준 없는 주관적 판단. "무엇을 기준으로 가장 우수한가"가 없으면 논문에서 삭제 대상.'
  },
  {
    text: '"레이어 수를 늘렸더니 정확도가 올랐다 (ablation 실험)."',
    type: 'fact',
    causal: true,
    feedback: '✅ 사실 + 인과 증거 — ablation은 인과 주장의 실험적 근거다. 다른 변수를 통제했다면 강한 주장이다.'
  },
  {
    text: '"정확도가 높은 모델일수록 파라미터 수가 많다."',
    type: 'interpretation',
    causal: false,
    feedback: '🔵 해석 + 상관관계 — 파라미터가 많아서 정확도가 높은 건지, 정확도가 높은 모델을 만들다 보니 파라미터가 늘어난 건지 알 수 없다. 인과 주장이 아니다.'
  },
  {
    text: '"학습률을 0.001로 설정했을 때 수렴 속도가 가장 빨랐다."',
    type: 'fact',
    causal: null,
    feedback: '✅ 사실 — 실험 설정과 관찰 결과. 이 조건에서의 사실. 단, 다른 조건에서의 일반화는 별도 실험 필요.'
  }
];

let currentIdx = 0;

function showClaim(idx) {
  const c = claims[idx];
  document.getElementById('claim-text').textContent = c.text;
  document.getElementById('claim-result').innerHTML = '';
  document.getElementById('claim-counter').textContent = `${idx+1} / ${claims.length}`;
}

function judgeClaim(userType) {
  const c = claims[currentIdx];
  const correct = userType === c.type;
  const resultEl = document.getElementById('claim-result');

  const typeLabel = {'fact': '사실', 'interpretation': '해석', 'opinion': '의견'};
  const causalLabel = c.causal === true ? ' · 인과 근거 있음' :
                      c.causal === false ? ' · 상관관계 (인과 아님)' : '';

  resultEl.innerHTML = `
    ${correct ? '✅ 정확!' : '❌ 다시 생각해보세요'}<br/>
    <strong>정답: ${typeLabel[c.type]}${causalLabel}</strong><br/>
    <span style="font-size:.85rem">${c.feedback}</span>
  `;
  resultEl.style.color = correct ? 'var(--accent3)' : 'var(--accent5)';
}

function nextClaim() {
  currentIdx = (currentIdx + 1) % claims.length;
  showClaim(currentIdx);
}

showClaim(0);
```

**HTML 구조 (calc 블록 내)**:
```html
<div id="claim-text" style="font-style:italic;margin-bottom:12px;"></div>
<div id="claim-counter" style="color:var(--muted);font-size:.8rem;margin-bottom:8px;"></div>
<div class="cr">
  <button class="btn btn-s" onclick="judgeClaim('fact')">사실</button>
  <button class="btn btn-s" onclick="judgeClaim('interpretation')">해석</button>
  <button class="btn btn-s" onclick="judgeClaim('opinion')">의견</button>
</div>
<div class="cres" id="claim-result"></div>
<button class="btn btn-s" onclick="nextClaim()" style="margin-top:8px;">다음 주장 →</button>
```

## 퀴즈

**질문**: 논문에서 "레이어를 제거하니 성능이 5% 하락했다 (ablation 실험)"는 어떤 종류의 주장인가?

| 선택지 | 정답 여부 |
|--------|----------|
| (a) 의견 — 저자의 주관적 판단이므로 | ❌ |
| (b) 상관관계 — 두 변수가 같이 변했으므로 | ❌ |
| (c) 인과 관계의 실험적 근거 — 변수를 통제하고 개입(removal)했으므로 | ✅ |
| (d) 해석 — 수치가 아닌 설명이므로 | ❌ |

**정답 위치**: (c)
**해설**: Ablation은 특정 요소를 제거(개입)하는 통제 실험이다. "제거했더니 성능이 내렸다"는 그 요소가 성능에 인과적으로 기여한다는 증거다. 상관은 관찰이고, ablation은 개입이다.
