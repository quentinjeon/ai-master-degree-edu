## EASY BOX 내용

**핵심 비유**: "시험 점수가 평균보다 1점 높아서 통계적으로 유의미하다 해도, 1점 차이가 입시에서 의미 있는가? 효과크기(Effect Size)는 '얼마나 다른가'를 본다."

**예시 리스트**:
- p < 0.05 but tiny effect: 10만 명 데이터에서 새 약이 혈압을 0.1mmHg 낮춤 → 통계적 유의미, 임상적 무의미
- 큰 효과, p > 0.05: 샘플 5명에서 큰 차이가 보였지만 p = 0.12 → 데이터가 더 필요한 것
- AI에서: "정확도 0.3% 향상 (p < 0.001)"은 상용 시스템에서 의미 있을 수도, 없을 수도 있다. Effect size + 비용이 함께 보고돼야 한다

### 섹션 1: 효과크기 (Effect Size) — 차이의 "크기"를 측정한다

**아이콘**: `📐`
**제목**: `통계적 유의미함 ≠ 실질적 중요함`

**설명**: p-value는 "차이가 우연이 아닐 확률"을 말합니다. 그러나 샘플이 매우 많으면 아주 작은 차이도 p < 0.05가 됩니다. 효과크기는 "차이가 실제로 얼마나 큰가"를 측정합니다. SCI 논문에서 두 가지를 함께 보고하는 것이 표준입니다.

**수식 박스 1**:
- 레이블: `Cohen's d — 두 집단 평균 차이의 효과크기`
- 수식:
```
d = \frac{\bar{X}_1 - \bar{X}_2}{s_{\text{pooled}}}
\quad\text{where}\quad
s_{\text{pooled}} = \sqrt{\frac{(n_1-1)s_1^2 + (n_2-1)s_2^2}{n_1+n_2-2}}
```
- 주석:
  - `X̄₁, X̄₂`: 두 그룹의 평균 (예: 모델 A vs 모델 B의 정확도)
  - `s_pooled`: 합동 표준편차 (두 그룹의 분산을 가중 평균)
  - d = 0.2: 작은 효과 / d = 0.5: 중간 / d = 0.8 이상: 큰 효과 (Cohen's 기준)
  - 단위 없음: 표준편차 단위로 표준화했으므로 서로 다른 스케일 비교 가능

**표 (효과크기 해석 기준)**:

| Cohen's d | 효과 크기 | 해석 | ML 논문 예시 |
|-----------|:-------:|------|------------|
| 0.0 ~ 0.2 | 매우 작음 | 실질적 의미 재검토 | 정확도 0.1% 향상 |
| 0.2 ~ 0.5 | 작음 | 주목할 만하나 제한적 | 정확도 1~2% 향상 |
| 0.5 ~ 0.8 | 중간 | 실질적으로 의미 있음 | 정확도 3~5% 향상 |
| 0.8 이상 | 큼 | 강한 실질적 효과 | 정확도 10% 이상 향상 |

**callout (warn)**:
- 내용: **p-value와 effect size를 모두 보고한다.** "p = 0.03, d = 0.15"는 통계적으로 유의미하지만 효과가 매우 작다는 뜻이다. "p = 0.08, d = 0.72"는 샘플이 부족해서 유의미성은 낮지만 효과는 크다는 뜻이다. 두 정보를 함께 봐야 한다.

### 섹션 2: 핵심 성능표 설계 — 논문 Table의 표준 구조

**아이콘**: `📊`
**제목**: `한 눈에 비교되는 성능표를 만드는 법`

**설명**: 논문의 Results 섹션에서 Table 1은 논문의 핵심 주장을 전달합니다. 좋은 성능표는 독자가 10초 안에 "제안 방법이 기준 대비 얼마나 좋은가"를 이해할 수 있어야 합니다.

**표 (성능표 설계 원칙)**:

| 원칙 | 잘못된 예 | 올바른 예 |
|------|---------|---------|
| Baseline 포함 | 제안 방법만 표시 | 기준 모델 + SOTA + 제안 방법 |
| 여러 Metric | 정확도만 | Accuracy + F1 + 추론 시간 |
| 통계적 표기 | 87.3% | 87.3 ± 0.4% |
| 강조 표시 | 없음 | **굵게** = 최고 성능 |
| 데이터셋 분리 | 하나만 | Dataset A / Dataset B 열 분리 |

**수식 박스 2**:
- 레이블: `성능 개선율 계산 표준`
- 수식:
```
\text{상대 개선율} = \frac{\text{제안} - \text{baseline}}{\text{baseline}} \times 100\%
\quad\text{vs}\quad
\text{절대 개선} = \text{제안} - \text{baseline}
```
- 주석:
  - baseline이 90%일 때 제안이 91%: 절대 개선 = 1%, 상대 개선 = 1.1%
  - baseline이 50%일 때 제안이 55%: 절대 개선 = 5%, 상대 개선 = 10%
  - 논문에서 둘 다 보고하는 것이 좋음 (상대 개선만 쓰면 baseline이 낮을 때 과장 가능)
  - 오류율이 중요한 경우: 오류율 감소율 = (baseline error - 제안 error) / baseline error

### 섹션 3: 세부 케이스 분석 — 평균이 가리는 진실

**아이콘**: `🔬`
**제목**: `전체 평균 뒤에 숨은 패턴을 찾는다`

**설명**: 전체 평균 정확도가 좋아도, 특정 클래스나 서브그룹에서는 나빠졌을 수 있습니다. SCI급 논문은 평균 외에 세부 분석을 요구합니다. 이 분석이 논문의 Discussion과 Limitation을 풍부하게 만듭니다.

**표 (세부 분석 4종류)**:

| 분석 유형 | 확인하는 것 | 발견 가능한 문제 |
|----------|-----------|--------------|
| **클래스별 분석** | 각 라벨에서의 Precision/Recall | 소수 클래스 성능 저하 |
| **서브그룹 분석** | 특정 속성(나이, 도메인 등)별 성능 | 편향(bias) 발견 |
| **오류 유형 분석** | False Positive vs False Negative | 어느 방향 오류가 더 위험한가 |
| **실패 사례 분석** | 틀린 예시 10~20개 직접 확인 | 모델의 체계적 약점 발견 |

**수식 박스 3**:
- 레이블: `불균형 클래스에서의 성능 평가`
- 수식:
```
\text{Macro-F1} = \frac{1}{K}\sum_{k=1}^K F1_k
\quad\text{vs}\quad
\text{Weighted-F1} = \sum_{k=1}^K \frac{n_k}{N} F1_k
```
- 주석:
  - K: 클래스 수, nk: k번째 클래스의 샘플 수, N: 전체 샘플 수
  - Macro-F1: 모든 클래스를 동등하게 취급 (소수 클래스 성능 중시)
  - Weighted-F1: 샘플 수에 비례하여 가중 (다수 클래스 성능 중시)
  - 논문에서 클래스 불균형이 있으면 두 가지 모두 보고해야 함

**callout (tip)**:
- 내용: **실패 사례를 논문에 공개하는 것은 강점이다.** "Figure 3: 대표적인 실패 사례"처럼 오답 예시를 보여주면, 리뷰어가 "이 저자는 자기 모델의 한계를 정확히 파악하고 있다"고 판단한다. 실패 사례 분석이 Limitation + 향후 연구로 자연스럽게 이어진다.

### 섹션 4: 비용-성능 트레이드오프 — 실용적 기여의 판단 기준

**아이콘**: `⚖`
**제목**: `성능만 보지 않는다 — 비용 대비 가치를 본다`

**설명**: SCI급 논문에서 "성능이 1% 올랐다"는 충분하지 않을 수 있습니다. 그 1%를 위해 계산 비용이 10배 늘었다면, 실용적 가치가 있는지 의문입니다. 비용-성능 트레이드오프 분석은 레벨 3 논문의 필수 항목입니다.

**표 (비용-성능 트레이드오프 분석 항목)**:

| 비용 항목 | 측정 방법 | 논문에서의 표기 |
|---------|---------|-------------|
| 추론 지연 (Latency) | ms/sample | "평균 추론 시간: 12ms" |
| 계산 복잡도 | FLOP 수 | "GFLOPs: 3.2 vs 1.8 (기준)" |
| 메모리 사용량 | GPU 메모리 (GB) | "학습 메모리: 24GB" |
| 학습 시간 | epoch당 시간 | "V100 1개에서 72시간" |
| 파라미터 수 | 백만 단위 (M) | "파라미터: 340M vs 110M" |

**수식 박스 4**:
- 레이블: `성능-비용 효율 지표 (Performance-Cost Ratio)`
- 수식:
```
\text{효율} = \frac{\Delta \text{Performance}}{\Delta \text{Cost}}
= \frac{\text{Acc}_{\text{제안}} - \text{Acc}_{\text{baseline}}}{\text{FLOPs}_{\text{제안}} / \text{FLOPs}_{\text{baseline}}}
```
- 주석:
  - 분자: 성능 향상량 (예: +2.3%)
  - 분모: 비용 증가 비율 (예: 1.5x = 50% 증가)
  - 효율이 높을수록 적은 비용으로 큰 성능 향상
  - 논문 Figure: 성능(y축) vs 계산 비용(x축) 산점도로 시각화

### 섹션 5: 일반화 가능성 — 내 결과는 어디까지 통하는가

**아이콘**: `🌐`
**제목**: `"이 데이터에서만"인가, "어디서나"인가`

**설명**: 연구의 일반화 가능성(generalizability)은 결과의 가장 중요한 측면 중 하나입니다. 내 방법이 특정 데이터셋·언어·도메인에서만 좋은지, 아니면 더 넓은 범위에서도 통하는지 논의해야 합니다.

**표 (일반화 가능성 검증 방법)**:

| 검증 유형 | 방법 | 논문에서의 의미 |
|---------|------|-------------|
| **다중 데이터셋** | 3개 이상 다른 데이터셋에서 실험 | "다양한 도메인에서 일관된 성능" |
| **Distribution shift** | 학습 분포와 다른 데이터로 테스트 | "OOD 강건성" |
| **크기 변화** | 더 큰/작은 모델로도 작동하는지 | "Scaling에 강건한 방법" |
| **언어/도메인 변환** | 영어 모델을 한국어에 적용 | "언어 독립적 방법" |

**callout (info)**:
- 내용: **일반화 가능성의 한계를 솔직하게 적는 것이 강점이다.** "본 연구는 의료 영상 도메인에서만 검증됐으며, 자연어 도메인으로의 직접 적용은 추가 실험이 필요하다"처럼 범위를 명확히 하면, 리뷰어가 "범위 밖의 결과를 주장한다"고 공격할 수 없다.

## 인터랙티브: 효과크기 계산기 + 비용-성능 트레이드오프

**제목**: `🔢 Cohen's d 계산기 + 성능-비용 효율 분석`

**파라미터**:

| ID | 이름 | min | max | step | default |
|----|------|-----|-----|------|---------|
| `mean1` | 제안 방법 평균 (%) | 50 | 100 | 0.1 | 87.3 |
| `mean2` | 기준 모델 평균 (%) | 50 | 100 | 0.1 | 85.0 |
| `std1` | 제안 방법 표준편차 | 0.1 | 5 | 0.1 | 0.4 |
| `std2` | 기준 모델 표준편차 | 0.1 | 5 | 0.1 | 0.6 |
| `n1` | 제안 방법 실험 수 | 1 | 20 | 1 | 5 |
| `n2` | 기준 모델 실험 수 | 1 | 20 | 1 | 5 |
| `flop_ratio` | 계산 비용 비율 (제안/기준) | 0.5 | 5 | 0.1 | 1.5 |

**JavaScript 로직**:
```javascript
function updateEffectSize() {
  const m1 = parseFloat(document.getElementById('mean1').value);
  const m2 = parseFloat(document.getElementById('mean2').value);
  const s1 = parseFloat(document.getElementById('std1').value);
  const s2 = parseFloat(document.getElementById('std2').value);
  const n1 = parseInt(document.getElementById('n1').value);
  const n2 = parseInt(document.getElementById('n2').value);
  const flopRatio = parseFloat(document.getElementById('flop_ratio').value);

  document.getElementById('mean1-v').value = m1.toFixed(1);
  document.getElementById('mean2-v').value = m2.toFixed(1);
  document.getElementById('std1-v').value = s1.toFixed(1);
  document.getElementById('std2-v').value = s2.toFixed(1);
  document.getElementById('n1-v').value = n1;
  document.getElementById('n2-v').value = n2;
  document.getElementById('flop_ratio-v').value = flopRatio.toFixed(1);

  // Cohen's d (pooled)
  const s_pooled = Math.sqrt(((n1-1)*s1*s1 + (n2-1)*s2*s2) / (n1+n2-2));
  const d = Math.abs(m1 - m2) / s_pooled;

  const dLevel = d < 0.2 ? '매우 작음 🔴' :
                 d < 0.5 ? '작음 🟡' :
                 d < 0.8 ? '중간 🟢' :
                           '큼 🏆';

  // 절대/상대 개선
  const absImprove = (m1 - m2).toFixed(2);
  const relImprove = ((m1 - m2) / m2 * 100).toFixed(2);

  // 비용-성능 효율
  const efficiency = Math.abs(m1 - m2) / flopRatio;

  // t-test 간단 버전 (풀링 SE)
  const se = s_pooled * Math.sqrt(1/n1 + 1/n2);
  const t = Math.abs(m1 - m2) / se;
  const df = n1 + n2 - 2;
  // p-value 근사 (간단한 t분포 근사)
  const pApprox = df > 30 ? (t > 1.96 ? '< 0.05' : '≥ 0.05') :
                  df > 10 ? (t > 2.23 ? '< 0.05' : '≥ 0.05') :
                             (t > 2.78 ? '< 0.05' : '≥ 0.05');

  document.getElementById('effect-res').innerHTML =
    `<strong>Cohen's d = ${d.toFixed(3)}</strong>  (효과 크기: ${dLevel})<br/>
     p-value 근사: ${pApprox} (t = ${t.toFixed(2)}, df = ${df})<br/>
     절대 개선: ${absImprove >= 0 ? '+' : ''}${absImprove}%<br/>
     상대 개선: ${relImprove >= 0 ? '+' : ''}${relImprove}%<br/>
     비용 증가: ${flopRatio}x<br/>
```

## 퀴즈

**질문**: 두 모델을 비교한 논문에서 "p = 0.001, Cohen's d = 0.18"이라면 어떻게 해석해야 하는가?

| 선택지 | 정답 여부 |
|--------|----------|
| (a) p < 0.05이므로 완전히 유의미한 결과다 | ❌ |
| (b) 통계적으로 유의미하지만 효과 크기가 매우 작아 실질적 의미가 제한적이다 | ✅ |
| (c) Cohen's d < 0.2이므로 이 실험은 실패다 | ❌ |
| (d) p-value와 Cohen's d는 같은 정보를 다르게 표현한 것이다 | ❌ |

**정답 위치**: (b)
**해설**: p = 0.001은 이 차이가 우연이 아닐 확률이 매우 높다는 뜻이다 (보통 대규모 샘플에서 발생). 그러나 Cohen's d = 0.18은 차이가 표준편차의 18%에 불과해 실질적으로 매우 작은 차이임을 뜻한다. 두 정보를 함께 보고하고 해석해야 한다.
